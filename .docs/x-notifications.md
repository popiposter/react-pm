# Notifications

## Текущее решение

В проекте больше не используется `@mantine/notifications`.

Для уведомлений используется `sonner`, а для действий, требующих осознанного выбора пользователя, применяются обычные dialog-сценарии на уровне приложения.

## Когда используем toast

- успешное сохранение
- ошибка сохранения
- запуск синхронизации
- результат заполнения демо-данными
- сообщения про login/logout flow

## Когда не используем toast

- конфликты данных, где нужен выбор пользователя
- потенциально опасные подтверждения
- длинные сценарии, где лучше явный banner или dialog

## Практическое правило

- `toast.success` — короткое подтверждение успешного действия
- `toast.error` — заметная, но не блокирующая ошибка
- `toast.loading` — краткое состояние фоновой операции

## Почему это решение выбрано

- хорошо сочетается с текущим UI shell
- простой API
- меньше инфраструктурного шума, чем у старой схемы с provider-спецификой Mantine
