# План интеграции с 1С

## Цель

Подключить 1С backend без переписывания экранов и без отказа от offline-first сценария.

## Что уже готово

- `OneCAuthTransport` stub
- `OneCSyncTransport` stub
- env-based factory для auth и sync transport
- auth session с token contract
- repository layer между UI и данными
- outbox и pending sync

## Рекомендуемый порядок интеграции

### Этап 1. Авторизация

Сделать в 1С endpoints:

- `POST /auth/login`
- `POST /auth/refresh`
- опционально `POST /auth/logout`

На фронте:

- реализовать `OneCAuthTransport.login()`
- реализовать `OneCAuthTransport.refresh()`
- пробросить backend errors в понятный UX

### Этап 2. Контекст пользователя

Нужно определить:

- employee id
- display name
- подразделение
- роли
- доступные проекты

Рекомендуемый результат:

- отдельный профиль пользователя после login
- claims/roles в session или profile response

### Этап 3. Чтение справочников

Сначала подключить только read-потоки:

- задачи
- проекты
- статусы табелей

Это позволит быстро проверить auth, routing и data loading без риска для сохранения.

### Этап 4. Сохранение табелей

Нужно согласовать DTO:

- идентификатор табеля
- дата
- версия или серверный ETag
- статус
- строки табеля
- состав строки: задача, время начала, время окончания, длительность, описание

Важно:

- фронту нужен versioning для conflict detection
- backend должен возвращать новый version после успешного сохранения

### Этап 5. Конфликты

Нужно заранее договориться о семантике:

- что считается конфликтом
- по какому признаку сервер возвращает `409`
- какие данные возвращаются для ручного разрешения конфликта

Минимально достаточный контракт:

- `409 Conflict`
- серверная версия табеля
- причина конфликта

### Этап 6. Фоновая синхронизация

После стабилизации обычного save flow:

- добавить background replay outbox
- подключить retry policy
- различать временные и постоянные ошибки

## Что не стоит делать

- не привязывать UI напрямую к 1С endpoints
- не убирать local repository до появления надежного sync flow
- не смешивать auth, read-model и save-model в один хаотичный adapter

## Роли

Сейчас роли не реализованы.

Рекомендуемый target:

- `user`:
  - ввод и редактирование своих табелей
- `manager`:
  - просмотр и подтверждение табелей команды
- `admin`:
  - управление справочниками, доступами и служебными операциями

На фронте это лучше закладывать через claims в session/profile и route guards, а не через разрозненные проверки в компонентах.
