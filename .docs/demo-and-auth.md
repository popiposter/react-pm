# Авторизация и демо-режим

## Что работает сейчас

В приложении уже есть отдельный экран логина и защищенные маршруты.

Маршруты:

- `/login`
- `/timesheets`
- `/timesheet/$date`

`/timesheets` и `/timesheet/$date` доступны только после входа.

## Демо-вход

По умолчанию на форме подставлены:

- логин: `demo.user`
- пароль: `demo`

Текущая demo-реализация принимает любую непустую пару логин/пароль.

После входа создается локальная сессия с:

- `user`
- `accessToken`
- `refreshToken`
- `expiresAt`
- `authStrategy`

## Поведение сессии

- сессия сохраняется локально
- при истечении срока действия выполняется logout с причиной `expired`
- если автообновление не удалось, выполняется logout с причиной `refresh-failed`
- login page умеет показывать баннер по причине редиректа

## Демо-данные

На экране логина есть кнопка заполнения локальной базы демо-данными.

Что добавляется:

- набор задач
- несколько табелей за последние дни
- разные статусы табелей

Эти данные записываются в локальное IndexedDB-хранилище и подходят для полноценного демо списка, редактора и sync queue.

## Что будет с auth позже

Переход на 1С будет идти через `OneCAuthTransport`.

План:

1. Login по паролю
2. Получение access token и refresh token
3. Автообновление сессии
4. Загрузка профиля и ролей
5. Ограничение маршрутов и действий по claims

## Есть ли админ / пользователь

Пока нет. Разделение ролей еще не внедрено.

Сейчас приложение считает, что работает один авторизованный пользователь.

Целевой вариант:

- `user`
- `manager`
- `admin`

Роли должны приходить с backend, а frontend будет использовать их для guards и feature access.
